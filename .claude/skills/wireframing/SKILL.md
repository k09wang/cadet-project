# Wireframing — 저충실도 화면 설계

자연어 브리프를 다운스트림 디자인이 소비할 수 있는 구조화된 와이어프레임 문서로 변환한다. 시각적 충실도가 아니라 **구조·흐름·데이터 의존성**을 정의하는 것이 목적이다.

## 왜 이 단계가 분리되어야 하는가

시각 디자인(색·타이포·간격)을 화면 구조와 동시에 결정하면, 구조 변경 시 디자인을 통째로 다시 해야 한다. 와이어프레임을 먼저 확정하면 디자인 단계는 "검증된 구조 위에 시각 언어만 입히는" 좁은 작업이 되어 재작업이 줄어든다.

## 작업 순서

1. **기존 자산 조사** — `src/app`(라우트)과 `src/components/{도메인}`을 Grep/Glob으로 훑어 재사용 가능한 화면·컴포넌트를 먼저 찾는다. artbridge 도메인: home, studio, posts, programs, community, contracts, applications, dashboard, creators, notification, membership.
2. **화면 분해** — 브리프를 라우트 단위 화면으로 쪼갠다.
3. **화면별 설계** — 아래 스키마대로 각 화면을 기술한다.
4. **흐름 연결** — 화면 간 이동(진입점, CTA, 성공/실패 분기)을 기술한다.
5. **데이터 의존성** — 각 화면이 읽고 쓰는 엔티티를 Prisma 모델명 기준으로 표기한다(예: Post, Program, ProgramApplication, Contract, MembershipPlan).

## 출력 스키마 — `_workspace/pipeline/01_wireframe.md`

```markdown
# Wireframe: {기능명}

## Assumptions
- (브리프가 모호할 때 가정한 내용. 없으면 생략)

## 화면 목록
| 화면 | 라우트 | 목적 | 재사용 컴포넌트 |
|------|--------|------|----------------|

## 화면 상세 (화면마다 반복)
### {화면명} ({라우트})
- **레이아웃 영역:** 헤더 / 본문 / 사이드 / 푸터 — 각 영역에 무엇이 들어가는가
- **컴포넌트:** [신규] / [재사용: 경로] 구분 목록
- **사용자 흐름:** 진입 → 행동 → 결과(성공/실패 분기)
- **상태:** 정상 / 빈(empty) / 로딩 / 에러 — 각각 무엇을 보여주는가
- **데이터 의존성:** 읽기[엔티티...] 쓰기[엔티티...]
- **근거:** 이 레이아웃을 택한 이유 한 줄

## 다음 단계 인계
- 디자인 단계(expert-frontend)가 시각 토큰을 입힐 우선순위 화면 명시
```

## 원칙

- **저충실도 고정**: 색상 코드, 폰트, px 간격을 쓰지 않는다. 영역과 위계만 표현한다.
- **상태 누락 금지**: 데이터를 표시하는 모든 화면은 빈/로딩/에러 상태를 반드시 기술한다. 실무 버그의 다수가 빈/에러 상태 누락에서 나온다.
- **재사용 표기 의무**: 신규 컴포넌트를 제안할 때마다 "기존에 유사한 것이 없는가"를 먼저 확인한 흔적을 남긴다.
- **이유 우선**: 규칙을 나열하기보다 결정 근거를 남겨, 디자인·구현 단계가 의도를 이해하고 엣지 케이스를 판단하게 한다.
