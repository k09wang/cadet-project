# SPEC-016: 역할별 GNB 이후 핵심 활동 허브와 결제/운영 동선 보완

> 기준 문서: `ArtBridge_유저플로우_확장안_2026-06-23.md`
> 선행 상태: SPEC-015에서 멤버십 결제 상태, 프로그램 선착순 결제 상태, 작품 주문/배송, 정산, 알림 이벤트의 데이터/도메인 기반을 구축했다.
> 목적: 역할별 GNB 개편 이후 팬과 크리에이터가 주요 업무를 빠짐없이 완료할 수 있도록 남은 화면, 진입점, 결제 확인, 운영 설정 동선을 보완한다.

## HISTORY

- 2026-06-23: 최초 작성. GNB 개편 이후 남은 보완점(팬 활동 허브, 작품 주문 목록, 정산 설정, checkout 전용 화면, 프로그램 FAQ, 알림 필터)을 실행 가능한 SPEC으로 정리.

## 1. 개요

현재 ArtBridge는 확장 유저플로우에 맞춰 프로그램 선착순 신청, 멤버십 결제, 작품 구매/배송, 크리에이터 정산 데이터의 핵심 기반을 갖췄다. 다만 사용자가 역할별 GNB를 통해 이동한 뒤 실제로 기대하는 업무 화면이 일부 부족하다.

본 SPEC은 새 도메인을 크게 추가하기보다, **이미 구현된 도메인 기능을 팬과 크리에이터가 자연스럽게 발견하고 완료할 수 있는 화면 흐름**을 보완한다.

## 2. 현재 확인된 갭

### 2.1 미인증 사용자

- 공개 GNB의 데스크톱 메뉴는 `작가 찾기`, `작품 보기`, `프로그램`, `이용 안내`로 구성되어 있으나, 모바일에서는 공개 탐색 메뉴가 충분히 노출되지 않는다.
- 미인증 사용자는 로그인 전에도 작가/작품/프로그램을 둘러볼 수 있어야 한다.

### 2.2 팬

- 팬 GNB의 `내 활동`은 여러 업무(프로그램 신청, 작품 주문, 멤버십, 결제 내역)를 포함해야 하지만 현재는 결제 내역 중심으로 보일 위험이 있다.
- 작품 주문 상세(`/artwork-orders/[id]`)은 있으나 팬이 전체 작품 주문을 모아 보는 전용 목록 동선이 부족하다.
- 작품 구매는 스튜디오 작품 카드 안에서 checkout 패널을 제공하지만, 일반 커머스 사용자가 기대하는 주문/배송/결제 확인 전용 화면으로 확장할 여지가 있다.
- 멤버십 가입과 프로그램 신청도 가격이 있는 경우 결제 전 확인 화면이 필요하다.

### 2.3 크리에이터

- 정산 목록(`/dashboard/creator/settlements`)은 있으나 정산을 받기 위한 계좌/사업자/지급 설정 화면이 없다.
- 작품 주문/배송 관리는 있으나 정산 보류 사유, 배송 문제, 환불 차감이 정산 설정과 연결되어 설명되지 않는다.
- 프로그램 상세에서 팬 문의를 줄이기 위한 FAQ/참여 전 안내 영역이 부족하다.

### 2.4 알림

- 알림 이벤트 종류가 늘었으므로 팬/크리에이터가 전체 알림을 필터링하거나 중요한 운영 알림을 놓치지 않도록 카테고리 구분이 필요하다.

## 3. 범위

### 포함

1. **공개 모바일 GNB 보완**
   - 미인증 사용자가 모바일에서도 작가/작품/프로그램/이용 안내로 이동할 수 있게 한다.

2. **팬 `내 활동` 허브 정리**
   - 팬의 프로그램 신청, 작품 주문, 멤버십, 결제 내역을 한 화면 또는 명확한 탭 구조로 연결한다.
   - 기존 `/dashboard/fan/payments`를 바로 버리지 않고, 현재 구현을 재사용해 전환 비용을 낮춘다.

3. **팬 작품 주문 목록**
   - 팬이 구매한 작품 주문을 목록으로 보고 주문 상태, 배송 상태, 송장 정보, 문제 신고/수령 확인 진입점을 확인한다.

4. **크리에이터 정산 설정**
   - 정산 받을 계좌/예금주/사업자 또는 개인 유형/검증 상태를 확인하고 수정하는 화면을 추가한다.
   - 실제 지급 자동화는 제외하되, 정산 불가 또는 보류 사유를 사용자에게 설명한다.

5. **결제 전 확인 화면**
   - 작품, 프로그램, 멤버십의 가격이 있는 구매/신청/가입 플로우에서 결제 전 주문 요약, 약관, 환불/취소 규정, mock escrow 결제 CTA를 제공한다.
   - 기존 즉시 처리 액션을 유지하되, UX는 전용 확인 단계로 이동한다.

6. **프로그램 FAQ/참여 전 안내**
   - 프로그램 상세에서 크리에이터가 정한 FAQ 또는 기본 안내를 노출해 팬이 신청 전 조건을 이해한다.

7. **알림 카테고리 필터**
   - 멤버십, 프로그램, 작품 주문/배송, 정산, 일반 알림을 구분해 볼 수 있게 한다.

### 제외

- 실제 PG사 결제수단 저장, 카드 토큰화, 정기결제 자동 청구.
- 실제 택배사 배송 조회 API 연동.
- 운영자 어드민의 정산 승인/반려 워크플로우 전체.
- 추천 알고리즘 고도화 또는 팬 취향 온보딩 전체 구현.
- 다중 배송지 주소록, 쿠폰/포인트, 부분 환불 자동화.

## 4. 기능 요구사항 (EARS)

### 공개 GNB

- **REQ-GNB-001**: WHEN 미인증 사용자가 모바일 화면에서 헤더를 보면, THE SYSTEM SHALL 작가 찾기, 작품 보기, 프로그램, 이용 안내로 이동할 수 있는 공개 탐색 진입점을 제공해야 한다.
- **REQ-GNB-002**: IF 사용자가 로그인하지 않았더라도, THE SYSTEM SHALL 공개 조회 가능한 작가/작품/프로그램 목록 접근을 막지 않아야 한다.

### 팬 활동 허브

- **REQ-FAN-001**: WHEN 팬이 GNB의 `내 활동`을 선택하면, THE SYSTEM SHALL 프로그램 신청, 작품 주문, 멤버십, 결제 내역을 한눈에 이동할 수 있는 활동 허브를 보여야 한다.
- **REQ-FAN-002**: IF 팬에게 진행 중인 작품 주문이 있으면, THE SYSTEM SHALL 주문 상태, 결제 상태, 배송 상태를 활동 허브 또는 작품 주문 목록에 표시해야 한다.
- **REQ-FAN-003**: WHEN 팬이 작품 주문 목록에서 주문을 선택하면, THE SYSTEM SHALL 기존 작품 주문 상세(`/artwork-orders/[id]`)로 이동해야 한다.
- **REQ-FAN-004**: IF 작품 주문이 배송 중 또는 배송 완료 후보 상태이면, THE SYSTEM SHALL 수령 확인과 문제 신고 진입점을 상태에 맞게 제공해야 한다.

### 크리에이터 정산 설정

- **REQ-PAYOUT-001**: WHEN 크리에이터가 GNB 또는 정산 화면에서 정산 설정을 선택하면, THE SYSTEM SHALL 정산 받을 계좌와 검증 상태를 확인할 수 있는 화면을 제공해야 한다.
- **REQ-PAYOUT-002**: IF 정산 설정이 미완료 상태이면, THE SYSTEM SHALL 정산 가능 금액이 있어도 지급 전 설정 완료가 필요하다는 안내를 제공해야 한다.
- **REQ-PAYOUT-003**: WHEN 크리에이터가 정산 계좌 정보를 저장하면, THE SYSTEM SHALL 입력값을 검증하고 저장 성공/실패 상태를 명확히 보여야 한다.

### 결제 전 확인

- **REQ-CHECKOUT-001**: WHEN 팬이 판매 작품의 `구매하기`를 선택하면, THE SYSTEM SHALL 수령자, 연락처, 배송지, 배송 요청사항, 작품 요약, 금액, 배송비, escrow 결제 CTA가 포함된 결제 전 확인 화면을 제공해야 한다.
- **REQ-CHECKOUT-002**: WHEN 팬이 유료 프로그램을 신청하면, THE SYSTEM SHALL 프로그램명, 일정, 정원/잔여석, 가격, 취소/환불 기준, 결제 CTA가 포함된 결제 전 확인 화면을 제공해야 한다.
- **REQ-CHECKOUT-003**: WHEN 팬이 유료 멤버십 가입을 선택하면, THE SYSTEM SHALL 플랜명, 혜택, 결제 주기, 금액, 해지 기준, 결제 CTA가 포함된 결제 전 확인 화면을 제공해야 한다.
- **REQ-CHECKOUT-004**: IF 결제 전 필수 정보가 비어 있으면, THE SYSTEM SHALL 주문 또는 신청을 생성하지 않고 필드 단위 오류를 표시해야 한다.

### 프로그램 FAQ

- **REQ-FAQ-001**: WHEN 팬이 프로그램 상세를 보면, THE SYSTEM SHALL 신청 CTA 이전 또는 인접 영역에 FAQ/참여 전 안내를 표시해야 한다.
- **REQ-FAQ-002**: IF 크리에이터가 별도 FAQ를 등록하지 않았다면, THE SYSTEM SHALL 가격, 일정, 취소, 준비물에 대한 기본 안내를 표시해야 한다.

### 알림

- **REQ-NOTI-001**: WHEN 사용자가 알림 화면에 진입하면, THE SYSTEM SHALL 전체, 멤버십, 프로그램, 작품, 정산 카테고리 필터를 제공해야 한다.
- **REQ-NOTI-002**: IF 알림 타입이 작품 배송 또는 정산 보류처럼 행동이 필요한 이벤트이면, THE SYSTEM SHALL 관련 상세 화면으로 이동할 수 있는 링크를 제공해야 한다.

## 6. 데이터 모델 영향

### 1차 구현

- 팬 작품 주문 목록은 기존 `ArtworkOrder`, `ArtworkShipment`, `ArtworkOrderIssue`, `Payment`를 재사용한다.
- 팬 활동 허브는 기존 `ProgramApplication`, `Membership`, `Payment`, `ArtworkOrder`를 조회한다.
- 프로그램 FAQ는 초기에는 저장 모델 없이 기본 안내 컴포넌트로 시작할 수 있다.
- 알림 필터는 기존 `Notification.type String`과 앱 레이어 타입 상수를 재사용한다.

### 2차 구현 후보

- 정산 설정을 영속화하려면 `CreatorPayoutAccount` 또는 `CreatorProfile` 확장 필드가 필요하다.
- 프로그램별 FAQ를 크리에이터가 직접 편집하려면 `Program.faqJson` 또는 `ProgramFaq` 모델이 필요하다.
- 주소록 저장을 도입하려면 `UserShippingAddress` 모델이 필요하다.

## 7. 구현 순서

1. **Phase 1: 네비게이션과 팬 활동 허브**
   - 공개 모바일 GNB 보완
   - 팬 `내 활동` 허브 또는 기존 결제 페이지의 활동 허브화
   - 팬 작품 주문 목록 추가

2. **Phase 2: 크리에이터 운영 설정**
   - 정산 화면에서 정산 설정 진입점 추가
   - 정산 설정 화면과 검증/저장 흐름 추가

3. **Phase 3: 결제 전 확인 UX**
   - 작품 checkout 전용 화면 또는 기존 인라인 패널의 route 분리
   - 프로그램 checkout 전용 화면
   - 멤버십 checkout 전용 화면

4. **Phase 4: 상세 보완**
   - 프로그램 FAQ/참여 전 안내
   - 알림 카테고리 필터
   - 반응형 QA와 라우트 연결 점검

## 8. 인수 기준

- **AC-001**: 미인증 모바일 사용자는 헤더에서 작가, 작품, 프로그램, 이용 안내로 이동할 수 있다.
- **AC-002**: 팬은 `내 활동`에서 프로그램 신청, 작품 주문, 멤버십, 결제 내역으로 이동할 수 있다.
- **AC-003**: 팬은 작품 주문 목록에서 각 주문의 상태, 배송 정보, 상세 링크를 확인할 수 있다.
- **AC-004**: 크리에이터는 정산 화면 또는 GNB에서 정산 설정 화면으로 이동할 수 있다.
- **AC-005**: 가격이 있는 작품/프로그램/멤버십 플로우는 결제 전 확인 단계에서 금액, 약관/취소 기준, 필수 입력값을 확인한다.
- **AC-006**: 프로그램 상세에는 FAQ 또는 기본 참여 전 안내가 표시된다.
- **AC-007**: 알림 화면은 주요 카테고리별 필터를 제공한다.
- **AC-008**: 관련 테스트, `tsc --noEmit`, ESLint가 통과한다.

## 9. 테스트 계획

- Header 컴포넌트 테스트: 미인증 모바일 공개 메뉴 렌더링.
- 팬 활동 허브 테스트: 활동 섹션 링크와 빈 상태/데이터 상태 렌더링.
- 팬 작품 주문 목록 쿼리 테스트: 본인 주문만 조회, 최신순 정렬, 배송/이슈 포함.
- 정산 설정 테스트: 권한 없는 사용자 차단, 필수 입력 검증, 저장 성공 상태.
- Checkout 테스트: 필수 배송/결제 확인값 누락 시 서버 액션 호출 방지.
- 프로그램 FAQ 테스트: 커스텀 FAQ 부재 시 기본 안내 노출.
- 알림 필터 테스트: 타입별 카테고리 매핑과 링크 노출.

## 10. 리스크와 결정 사항

- **결정 1**: 1차 구현에서는 새 데이터 모델을 최소화하고 기존 SPEC-015 모델을 재사용한다.
- **결정 2**: 정산 설정 영속화는 구현 시점에 `CreatorPayoutAccount` 신설과 `CreatorProfile` 확장 중 하나를 선택한다. 정산 계좌는 향후 민감정보 처리가 필요하므로 실제 계좌번호 저장 방식은 별도 보안 검토가 필요하다.
- **결정 3**: checkout 전용 화면은 UX상 필요하지만, 현재 인라인 작품 구매 패널이 이미 동작하므로 작품 checkout은 2차에서 라우트 분리해도 된다.
- **리스크 1**: 결제 전 확인 화면을 모두 route로 분리하면 서버 액션과 redirect 테스트 범위가 넓어진다.
- **리스크 2**: 팬 활동 허브를 새 route로 만들지 기존 `/dashboard/fan/payments`를 확장할지에 따라 GNB 링크와 문서 표현이 달라질 수 있다.
- **리스크 3**: 프로그램 FAQ를 모델 없이 기본 컴포넌트로 시작하면 크리에이터 편집 기능은 후속 SPEC으로 남는다.
