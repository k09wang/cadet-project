## 0. 문서 목적  
이 문서는 ArtBridge MVP를 **Codex / Claude Code 같은 agentic coding tool**로 빠르게 구현하기 위한 PRD입니다.  
이 문서는 단순히 사람이 읽는 기획서가 아니라, AI 코딩 에이전트에게 작업을 위임하기 위한 **작업 지시서 + 제품 명세 + 검증 기준**입니다.  
이 문서는 단순히 사람이 읽는 기획서가 아니라, AI 코딩 에이전트에게 작업을 위임하기 위한 **작업 지시서 + 제품 명세 + 검증 기준**입니다.  
* Codex: 레포지토리 구조를 읽고, 필요한 파일을 수정하고, 테스트/빌드 명령을 실행하는 작업 에이전트로 사용  
* Claude Code: CLAUDE.md 기반의 프로젝트 컨텍스트 유지, 코드 수정, 리팩터링, 검증 도구로 사용  
* 작업 단위: 개인별 역할 배분이 아니라 **기능 티켓 단위**로 분리  
* 개발 기준: “완벽한 상용 서비스”보다 **배포 환경에서 끊기지 않는 종단간 데모** 우선  
  
## 1. 제품 정의  
## 1.1 제품명  
**ArtBridge**  
가칭입니다. 최종 서비스명이 바뀌면 코드 식별자는 유지하고, 사용자 노출 카피만 find-replace 합니다.  
## 1.2 한 줄 정의  
ArtBridge는 신진작가가 자신의 창작 스튜디오를 개설하고, 팬이 멤버십·유료 포스트·클럽/프로그램을 통해 작가를 후원하고 교류할 수 있는 **창작자-팬 후원 커뮤니티**입니다.  
## 1.3 제품 콘셉트  
Steadio는 크리에이터의 지속 가능한 창작 활동을 위해 다음 기능 축을 제공합니다.  
* 크리에이터 페이지 개설  
* 월 정기 멤버십 생성/운영  
* 유료 포스트 발행  
* 클럽 생성 및 운영  
* 커뮤니티 기능  
* 멤버 관리  
* 메시지, 쿠폰, 오디오 플레이어, 배송지 수집 등 확장 기능  
ArtBridge는 이 중 과제 기간 안에 구현 가능한 핵심만 추려 다음 구조로 만듭니다.  
작가 스튜디오 → 멤버십 → 유료/멤버 전용 포스트 → 클럽/프로그램 → 팬 참여 신청 → 수락/결제 → 커뮤니티 → 리뷰  
작가 스튜디오 → 멤버십 → 유료/멤버 전용 포스트 → 클럽/프로그램 → 팬 참여 신청 → 수락/결제 → 커뮤니티 → 리뷰  
## 1.4 RFP 대응 원칙  
원래 RFP는 의뢰인과 프리랜서를 연결하는 WorkBridge형 외주 매칭 플랫폼입니다.  
ArtBridge는 이를 그대로 버리지 않고, 다음처럼 도메인을 변환합니다.  

| RFP 구조   | ArtBridge 변환                  |
| -------- | ----------------------------- |
| 의뢰인      | 팬 / 후원자 / 참여자                 |
| 프리랜서     | 크리에이터 / 신진작가                  |
| 프로젝트     | 클럽 / 프로그램 / 멤버십 오퍼링           |
| 지원       | 팬의 참여 신청 또는 멤버십 가입            |
| 지원 수락/거절 | 크리에이터의 신청 승인/거절               |
| AI 단가 분석 | AI 가격·혜택·프로그램 구성 추천           |
| 계약       | 프로그램 참여 계약서 / 멤버십 약관          |
| 결제/에스크로  | 팬의 선결제, 플랫폼 보관 상태, 완료 후 정산 처리 |
| 리뷰       | 팬 후기 + 크리에이터/팬 상호 평가          |
| 알림       | 신청, 수락, 거절, 마감, 결제/리뷰 요청 알림   |

## 2. 핵심 문제와 사용자  
## 2.1 문제 정의  
신진작가는 팬과 소통할 채널이 흩어져 있고, 지속 수익을 만들기 어렵습니다. SNS는 노출과 홍보에는 좋지만 멤버십, 유료 콘텐츠, 커뮤니티, 프로그램 운영을 한곳에서 관리하기 어렵습니다.  
팬은 좋아하는 작가를 후원하고 더 깊게 연결되고 싶지만, 후원·콘텐츠 구매·프로그램 참여·커뮤니티 활동이 분산되어 있어 작가와 지속적으로 관계를 맺기 어렵습니다.  
ArtBridge는 작가의 창작 활동을 하나의 스튜디오에 모으고, 팬이 멤버십·유료 포스트·클럽/프로그램으로 참여할 수 있게 만들어 창작자의 지속 가능한 수익화와 커뮤니티 빌딩을 돕습니다.  
## 2.2 사용자 유형  
## Creator / Artist  
신진작가, 일러스트레이터, 에세이스트, 독립출판 작가, 뮤지션, 공예 작가 등 팬과의 관계를 수익과 커뮤니티로 연결하고 싶은 사용자입니다.  
주요 행동:  
* 스튜디오 페이지 생성  
* 자기소개, 링크, 포트폴리오 등록  
* 멤버십 플랜 개설  
* 유료/멤버 전용 포스트 발행  
* 클럽/챌린지/클래스 개설  
* 팬 신청 수락/거절  
* 멤버와 커뮤니티에서 소통  
* 프로그램 완료 후 리뷰 확인  
## Fan  
좋아하는 작가를 후원하고, 콘텐츠를 구매하거나 프로그램에 참여하고 싶은 사용자입니다.  
주요 행동:  
* 작가와 프로그램 탐색  
* 멤버십 가입  
* 멤버 전용 포스트 확인  
* 클럽/프로그램 참여 신청  
* 결제 및 계약 동의  
* 커뮤니티 글 작성  
* 프로그램 완료 후 리뷰 작성  
  
## 3. MVP 성공 기준  
## 3.1 최종 데모 플로우  
최종 발표에서는 아래 흐름이 끊기지 않아야 합니다.  
1. 랜딩 페이지에서 ArtBridge의 가치 제안을 보여준다.  
2. 크리에이터로 로그인한다.  
3. 크리에이터 스튜디오를 확인하거나 수정한다.  
4. 멤버십 플랜을 생성하거나 확인한다.  
5. 멤버 전용 포스트를 작성한다.  
6. 클럽/프로그램을 생성한다.  
7. 팬으로 로그인한다.  
8. 팬이 작가 스튜디오와 프로그램을 탐색한다.  
9. 팬이 클럽/프로그램에 참여 신청한다.  
10. 크리에이터가 팬 신청을 수락한다.  
11. 팬에게 수락 알림이 표시된다.  
12. 팬이 계약/약관에 동의하고 결제를 진행한다.  
13. 결제 완료 후 팬은 멤버/참여자 전용 콘텐츠와 커뮤니티에 접근한다.  
14. 크리에이터가 멤버 또는 참여자 목록을 확인한다.  
15. 프로그램 완료 처리 후 리뷰를 작성한다.  
## 3.2 데모 최소 성공선  
시간이 부족할 경우 아래 8개만 살아 있으면 제출 가능한 MVP로 판단합니다.  
* Mock 또는 실제 로그인으로 Creator/Fan 전환 가능  
* 크리에이터 스튜디오 페이지 표시  
* 멤버십 플랜 표시  
* 멤버 전용 포스트 잠금/공개 처리  
* 클럽/프로그램 생성 및 상세 표시  
* 팬의 참여 신청  
* 크리에이터의 수락/거절 + 알림  
* 결제 Mock 또는 sandbox 상태 전환 후 리뷰 작성

## 5. 기술 스택  
## 5.1 권장 스택  

| 영역 | 선택 | 이유 |
| --------- | ------------------------------------------------- | ----------------------- |

| DB | PostgreSQL | 배포/평가 친화적 |
| ORM | Prisma | 스키마 기반 구현이 AI agent에 적합 |
| Auth | Auth.js 또는 Mock Auth | 시간에 따라 선택 |
| AI | OpenAI API JSON schema | 구조화된 추천 결과 |
| Payment | Mock provider 우선, PortOne/Toss sandbox adapter 선택 | 데모 안정성 우선 |
| Deploy | Vercel | Next.js 배포 최적 |
| Storage | URL 입력 우선, 시간 남으면 Cloudinary/Supabase Storage | 이미지 업로드 리스크 감소 |
  
****5.2 구현 선택 기준****  
* 시간이 1일 이상 남고 인증이 안정적이면 Auth.js를 적용한다.  
* 시간이 부족하면 MockAuthProvider와 seed user로 역할 전환을 구현한다.  
* 결제는 PaymentProvider 인터페이스를 먼저 만들고, 기본 구현체는 MockPaymentProvider로 둔다.  
* 실제 PG sandbox는 시간이 남을 때만 PortOnePaymentProvider 또는 TossPaymentProvider로 확장한다.  
* 모든 외부 키는 .env에 두고 .env.example을 반드시 만든다.

## 7. 데이터 모델 초안 — Prisma 기준  
아래 스키마는 agent가 구현을 시작할 때 기준으로 삼는 초안입니다. 기존 레포에 스키마가 있으면 먼저 비교한 뒤, 필요한 범위만 반영합니다.  
```
enum Role {
  FAN
  CREATOR
}

enum ProgramStatus {
  DRAFT
  RECRUITING
  CLOSED
  CONTRACTING
  IN_PROGRESS
  COMPLETED
  CANCELLED
}

enum ApplicationStatus {
  PENDING
  ACCEPTED
  REJECTED
  AUTO_REJECTED
  CANCELLED
}

enum PostAccessLevel {
  PUBLIC
  MEMBERS_ONLY
  PAID
}

enum MembershipStatus {
  ACTIVE
  CANCELED
}

enum ContractStatus {
  PENDING
  FAN_SIGNED
  CREATOR_SIGNED
  SIGNED
  CANCELLED
}

enum PaymentStatus {
  PENDING
  PAID
  RELEASED
  FAILED
  REFUNDED
}

enum NotificationType {
  APPLICATION_CREATED
  APPLICATION_ACCEPTED
  APPLICATION_REJECTED
  APPLICATION_AUTO_REJECTED
  PROGRAM_CLOSED
  PAYMENT_COMPLETED
  REVIEW_REQUESTED
}

model User {
  id           String   @id @default(cuid())
  email        String   @unique
  passwordHash String?
  name         String
  role         Role
  avatarUrl    String?
  bio          String?
  createdAt    DateTime @default(now())
  updatedAt    DateTime @updatedAt

  creatorProfile CreatorProfile?
}

model CreatorProfile {
  id              String   @id @default(cuid())
  userId          String   @unique
  user            User     @relation(fields: [userId], references: [id])
  studioName      String
  category        String
  introduction    String
  coverImageUrl   String?
```

## 19. 발표에서 방어할 의사결정  
## 19.1 왜 WorkBridge를 그대로 만들지 않았나?  
RFP의 핵심은 특정 도메인 그 자체보다 양면형 거래 플랫폼의 전체 흐름을 구현하는 것입니다. ArtBridge는 그 구조를 유지하되, 팀의 관심 도메인인 신진작가와 팬 커뮤니티로 전환했습니다.  
## 19.2 왜 실제 반복 결제를 구현하지 않았나?  
반복 결제와 실제 정산은 장애 리스크가 높고, 짧은 과제 기간에 비해 학습 대비 비용이 큽니다. 대신 Payment 모델, 결제 상태, 수수료 계산, 정산 상태 전환을 구현해 서비스 흐름을 검증했습니다.  
## 19.3 왜 메시지, 쿠폰, 오디오, 배송지 수집을 제외했나?  
Steadio 벤치마킹상 확장 기능으로 의미는 있지만, 이번 MVP의 핵심 흐름은 작가 스튜디오, 멤버십, 유료 포스트, 프로그램, 커뮤니티입니다. 확장 기능은 발표의 로드맵으로 분리했습니다.  
## 19.4 왜 Mock을 허용했나?  
목표는 상용 결제 시스템이 아니라, 제한된 시간 안에서 종단간 사용자 흐름을 배포 환경에서 안정적으로 증명하는 것입니다. Mock은 기능의 비즈니스 의미를 보존하면서 장애 리스크를 줄이는 선택입니다.  
  
## 20. 마지막 원칙  
이번 프로젝트에서 좋은 기능은 “멋진 기능”이 아니라 **데모를 살리는 기능**입니다.  
판단이 애매하면 아래 질문으로 결정합니다.  
1. 최종 데모 경로에 직접 필요한가?  
2. 구현하지 않으면 발표가 무너지는가?  
3. Mock으로 대체해도 비즈니스 의미가 전달되는가?  
4. 오늘 안에 붙이고 검증할 수 있는가?  
5. 배포 환경에서 안정적으로 동작할 가능성이 높은가?  
하나라도 명확하지 않으면 만들지 않습니다.
